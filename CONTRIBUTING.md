# Руководство по участию в проекте

Спасибо за интерес к **Awesome DFIR**! Любой вклад приветствуется — от исправления опечатки до добавления целого нового раздела.

---

## Способы участия

- **Исправить ошибку** — некорректная команда, устаревший путь, неверная информация
- **Добавить артефакт** — описание нового forensic-артефакта по шаблону
- **Улучшить описание** — сделать раздел понятнее, добавить примеры
- **Добавить инструмент** — ссылка на утилиту с кратким описанием
- **Разработать новый раздел** — Cloud, Containers, Kubernetes, Network
- **Перевод** — английская версия документации

---

## Процесс внесения изменений

1. **Создайте Issue** (необязательно для мелких правок)

   Для крупных изменений рекомендуем сначала обсудить через [Issue](https://github.com/st3l1n/awesome_dfir/issues), чтобы убедиться, что направление соответствует проекту.

2. **Сделайте Fork** репозитория

3. **Создайте ветку** с понятным названием:
   ```
   git checkout -b fix/windows-registry-path
   git checkout -b add/linux-auditd-artifact
   git checkout -b improve/macos-persistence-section
   ```

4. **Внесите изменения**, соблюдая структуру документов (см. ниже)

5. **Создайте Pull Request** с описанием изменений

---

## Структура документов

Каждый артефакт должен следовать единому шаблону:

```markdown
### Название артефакта [🔴/🟡/🟢] [⚠️] [⏱️]

**Что это:** Краткое описание артефакта и его назначение.

**Пути:**
- `/путь/к/файлу`
- `HKEY\Registry\Key` (для Windows)

**На какие вопросы отвечает:**
- Вопрос 1
- Вопрос 2

**Парсинг:**
```bash
# Комментарий с пояснением
команда --опция
```

**Инструменты:** [Tool Name](https://link), другой инструмент
```

### Система иконок

| Иконка | Значение |
|--------|----------|
| 🔴 | Критичный — собирать в первую очередь |
| 🟡 | Важный — стандартный IR |
| 🟢 | Дополнительный — при детальном расследовании |
| ⚠️ | Требует прав администратора / root |
| ⏱️ | Волатильный — исчезает после перезагрузки |
| 🔄 | Ротируется — может быть перезаписан |
| 🔐 | Может быть зашифрован |

## Требования к командам

- Команды должны быть **проверены** на указанной версии ОС
- Указывайте **версию ОС**, если команда работает только на определённых версиях
- Для дистрибутивов Linux используйте метки: `[deb]`, `[rpm]`, `[all]`
- Для Windows указывайте диапазон версий: `Windows 10+`, `Server 2016+`

---

## Добавление нового инструмента

При добавлении инструмента в раздел "Инструменты" или в Приложение 3:

- Указывайте ссылку на официальный репозиторий или документацию
- Добавляйте краткое описание (1 строка): что делает, для чего применяется в DFIR
- Для коммерческих инструментов обозначайте это явно

---

## Стиль написания

- Язык документации: **русский**
- Технические термины (имена команд, пути, названия инструментов) — без перевода
- Описания — понятные и лаконичные, без излишней воды
- Для форматирования используйте GitHub Flavored Markdown
